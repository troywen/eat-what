# 《等一下吃什么》架构设计文档

> 版本 v0.1 · Demo 阶段 · 设计原则：第一性原理 + 模块化 + 可迁移

---

## 1. 第一性原理分析：「吃什么」决策的本质

剥掉所有产品外壳，用户面临的是一个 **多约束下的进食方案选择问题**：

> 在给定资源与状态下，从候选集合中选出一个方案，使 **总效用最大**。

**总效用（Utility）** 由三个基本要素决定：

| 基本要素 | 含义 | 对应输入信号 |
|---|---|---|
| **生理适配** | 身体在此刻需要什么（能量、温度、轻重） | 时间（早/午/晚/夜宵）、天气（温度/湿度/晴雨） |
| **心理适配** | 情绪想要什么补偿（治愈、庆祝、凑合） | 心情 |
| **成本约束** | 我愿意付出多少（时间、钱、精力、食材浪费） | 现有食材、模式（做/外卖/外出）、可用时间 |

由此推出三条互斥的解决路径（用户二选一之前的上层决策）：

1. **自己做饭** —— 资源利用最大化，成本最低，满足感来自"刚好用掉现有食材"
2. **外卖** —— 时间成本最小化，满足感来自"快 + 想吃"
3. **外出吃** —— 体验最大化，满足感来自"氛围 + 热门 + 顺路"

**推荐算法 = 对每条路径的候选集打效用分，取 Top-N。**
打分维度与第一性原理三要素一一对应，不允许出现"说不清为什么存在"的打分项。

## 2. 系统分层（可迁移性设计）

```
┌─────────────────────────────────────────────┐
│  UI 层 (src/sections, src/pages)             │  ← 可替换：Web / RN App / 小程序
├─────────────────────────────────────────────┤
│  核心层 (src/core/)  ★ 纯 TypeScript，零 React 依赖 ★ │  ← 原样搬到 Node 服务器 / App
│  ├── types.ts        领域模型                  │
│  ├── data/           数据层（菜谱/食材/商户）    │
│  ├── services/       外部服务抽象（接口+Mock+真实）│
│  └── engine/         推荐引擎（打分器管道）      │
└─────────────────────────────────────────────┘
```

**迁移路径**：`src/core` 不 import 任何 React / DOM API，未来可直接：
- 抽到独立 npm 包，服务端 `import { recommend } from 'eat-what-core'`
- 包进 React Native / Tauri 成为 App
- 数据层从本地 TS 数据换成数据库，只需替换 `data/` 的读取实现（通过 Repository 接口隔离）

## 3. 推荐引擎：打分器管道（Scorer Pipeline）

**核心思想**：每个打分维度是一个独立、可插拔、可单测的 **Scorer**；
最终得分 = Σ (权重ᵢ × 维度分ᵢ)，权重集中在 `weights.ts` 一处调节。

```
候选菜谱 ──▶ ┌ ingredientMatch  食材匹配度（缺几个料/覆盖率）
             ├ weatherFit       天气适配（寒冷→热汤锅物/炎热→凉拌清爽）
             ├ timeFit          时间适配（餐段匹配 + 烹饪时长 ≤ 可用时间）
             ├ moodFit          心情适配（心情→口味/菜型映射表）
             ├ seasonFit        季节适配（时令菜优先）
             ├ trendFit         热度分（近期热门加权）
             └ novelty          多样性（最近吃过降权，Demo 用会话内历史）
                   │
                   ▼  score = Σ wᵢ·sᵢ  ──▶ 排序取 Top-N，附「推荐理由分解」
```

- 每个 Scorer 输出 **0~1 标准化分数 + 人类可读理由**，理由直接展示给用户（可解释性）
- 新增维度 = 新增一个 scorer 文件 + 在管道注册 + 配权重，**不改任何现有代码**（开闭原则）
- 外卖/外出复用同一管道，只是候选集换成 `PlaceCandidate`，scorer 换成（距离/热度/时令/心情场景）

## 4. 外部服务抽象（services/）

Demo 阶段全部提供 Mock 实现 + 真实接口预留，接口先行：

| 服务 | 接口 | Demo 实现 | 生产实现 |
|---|---|---|---|
| `WeatherService` | `getCurrent(): Weather` | 按月份/小时启发式模拟，可手动切换 | Open-Meteo（默认，免 Key）/ 高德（有 Key 切换） |
| `VideoService` | `getGuides(dish): VideoGuide[]` | 生成 B站/抖音搜索直达链接 | 接入开放平台搜索 API |
| `AmapService` | `getStreetRank(city): Place[]` | 内置「扫街榜」Mock 数据 + 高德 URI 跳转 | 生产环境经自有服务端调用高德 POI API，Key 不下发前端 |

视频链接采用搜索直达 URL（`search.bilibili.com/all?keyword=xx做法`、`douyin.com/search/xx做法`），无需 API Key 即可用。

## 5. 数据层

- `ingredients.ts`：常见食材按类分组（肉蛋/蔬菜/豆制品/主食/菌菇/水产…），供点选
- `recipes.ts`：**≥ 24 道菜**，每道含：食材及精确用量（克/勺）、分步做法、烹饪时长、餐段、口味标签、温凉属性、难度、适合心情、时令月份、热度分
- `places.ts`：外卖/外出候选，含品类、人均、热度、时令标签、适合场景

## 6. 测试策略

- 引擎全部单测（vitest）：每个 scorer 的边界值、管道权重正确性、Top-N 排序、食材匹配缺料计算、模式切换路由
- `npm run build`（tsc 严格模式）必须零错误
- 人工审查清单：无 React 依赖渗入 core/、权重和为 1、推荐理由非空

## 7. 模块调整指南（给未来的自己）

| 想改什么 | 改哪里 |
|---|---|
| 调推荐口味 | `core/engine/weights.ts` |
| 加打分维度 | 新建 `core/engine/scorers/xxx.ts` → 注册进 `pipeline.ts` |
| 加菜谱 | `core/data/recipes.ts` |
| 接真实天气 | 已接入 Open-Meteo（`core/services/openMeteo.ts`）；换别的源就实现 `WeatherService` 接口 |
| 接高德 API | 实现 `AmapService` 接口替换 mock |
